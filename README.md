# Hash Table

Проект посвящён исследованию `хэш-таблицы` с бакетами из [`cache-friendly list`](https://github.com/BeVa733/soa_list) с разрешением коллизий с последующей оптимизацией операции поиска.

Работа разделена на два этапа:

1. **Исследование различных хэш-функций** по качеству распределения слов по бакетам.
2. **Оптимизация поиска в хэш-таблице** на основе профилирования с использованием различных низкоуровневых механизмов оптимизации (`intrinsics`, `inline asm`, `asm function`).

## Быстрый запуск

Сборка из корневой директории:

```bash
make
```

Требуемое окружение:
* `g++`
* `nasm`

В текущем `Makefile` собираются две программы:

```bash
./hash_research.out <normalized_file> [table_size]
./hash_opt.out <table_file> <query_file> [table_size]
```

Примеры запуска:

```bash
./hash_research.out norm_text.bin 4001
./hash_opt.out norm_text.bin norm_test.bin 17117
```

## Подготовка данных

В репозитории присутствует скрипт `get_text.sh`, который скачивает корпус текстов для тестирования.

Для финальной версии оптимизаций используется Python-скрипт `py_scripts/normalizer.py`. Он:
* переводит текст в нижний регистр;
* оставляет только символы `a-z` и апостроф;
* разбивает текст на слова;
* сохраняет слова в бинарный файл;
* записывает каждое слово в фиксированном формате по 32 байта (если слово короче - оно дополняется нулями).

Такой формат был введён на позднем этапе проекта для ускорения сравнения строк и упрощения SIMD-обработки.

## Этап 1. Исследование хэш-функций

Для исследовательской части используется `research_main.cpp`.

На этом этапе таблица многократно заполняется одним и тем же набором слов, но каждый раз с новой хэш-функцией. После заполнения фиксируется, сколько элементов попало в каждый бакет.

Программа поочерёдно тестирует несколько хэш-функций:

* `one_hash`
* `first_ascii_hash`
* `lenth_hash`
* `ascii_sum_hash`
* `hash_rol_xor`
* `gnu_hash`
* `crc32c_hash`

В набор включены как заведомо слабые функции, так и более содержательные варианты. Это нужно для того, чтобы увидеть разницу между плохим и хорошим хэшированием на одном и том же входе.

Для каждой функции создаётся CSV-файл с количеством элементов в каждом бакете. Эти файлы сохраняются в `results/tables`. Далее по ним строятся гистограммы распределения, которые сохраняются в `results/histograms`. За счёт этого результаты можно анализировать не только численно, но и визуально.

Исследовательская часть позволяет:

* сравнить, насколько равномерно разные функции распределяют слова по бакетам;
* увидеть, какие функции быстро приводят к длинным цепочкам и росту числа коллизий;
* выбрать хэш-функцию, которую имеет смысл использовать в оптимизационной части проекта.

Главный практический результат этого этапа состоит в том, что примитивные функции дают заметно более плохое распределение, тогда как более сильные функции распределяют элементы по таблице существенно ровнее. Это напрямую влияет на среднюю длину цепочек и, следовательно, на стоимость поиска.

## Этап 2. Оптимизация поиска

Во второй части проекта рассматривается отдельный сценарий: таблица заполняется одним набором слов, после чего выполняется большое число запросов поиска из второго набора. Для этого используется `optimization_main.cpp`.

В оптимизационной версии в качестве основной хэш-функции выбрана `crc32c_hash`, а сама операция `find_in_table` рассматривается как критический участок программы.

## Последовательность оптимизаций

Ниже приведена последовательность этапов оптимизации программы. Каждый раз для оптимизациии выбиралась функция определяемая профилировщиком как узкое место (`bottleneck`)

### 1. Базовая версия

Исходная версия использует обычную реализацию поиска по бакету и отдельный вызов хэш-функции.

### 2. Ускорение хэширования через intrinsics

Функция `crc32c_hash` реализована через `_mm_crc32_u8`, то есть через аппаратную инструкцию CRC32, доступную через intrinsics.

### 3. Дополнительные точечные оптимизации

После ускорения хэш-функции повторное профилирование показало, что распознавание слов сатло узким местом. Поэтому распознавание слов было выполнено с помощью `SIMD` инструкций в виде `inline assembler`.

### 4. Вынесение горячего пути в ассемблер

Следующим этапом стала большая ассемблерная функция `find_in_table.asm`.

В ней объединены несколько оптимизаций:
* вычисление `crc32c` прямо внутри поиска, вместо долгого вызова;

* вычисление индекса бакета без инструкции div (замена на умножение и сдвиг);

* проверка первого символа (`precheck`), позволяющая избавиться от лишних накладных расходов на вызов библиотечной функции strcmp;

Это позволило сократить накладные расходы на вызовы отдельных функций и избавиться от дорогостоящей операции `div`.

### 5. Оптимизация сравнения строк

На финальном этапе стандартное сравнение строк было заменено на собственную функцию `strcmp_32`, написанную на ассемблере.

Она загружает сразу 32 байта слова в `ymm`-регистр, сравнивает два блока через AVX-инструкции и проверяет, совпали ли все байты.

Такой подход стал возможен после перехода к фиксированному 32-байтному формату слова при нормализации текста.

## Результаты оптимизации

В каталоге `profiling_files` сохранены несколько последовательных запусков профилировщика. В качестве метрики использовалось количество выполненных инструкций `Ir` из Callgrind.

| Этап | Файл профилирования | Ir |
|---|---|---:|
| Базовая версия | `first.136720` | 714,468,687 |
| Hash через intrinsics | `second.143098` | 383,790,882 |
| Inline asm оптимизация| `third.153240` | 290,041,363 |
| Оптимизация поиска на asm | `fourth.69636` | 261,085,931 |
| Финальная версия до нового формата | `fifth_final.74730` | 258,740,242 |

Относительно первой версии итоговая конфигурация уменьшила число выполняемых инструкций примерно на **70.5%** и дала суммарное ускорение примерно в **3.39 раза**.


## Краткий вывод работы

Работа показала, что производительность хэш-таблицы определяется двумя факторами.

На первом этапе стало видно, что плохая хэш-функция сама по себе создаёт неравномерное распределение, увеличивает длину цепочек и ухудшает поиск.

 На втором этапе было показано, что после выбора более удачной функции заметный выигрыш дают уже машинно-ориентированные оптимизации горячего пути.

Итог проекта состоит в том, что сначала нужно обеспечить качественное распределение элементов по таблице, а затем уже оптимизировать конкретные узкие места реализации. Только сочетание этих двух этапов даёт заметный итоговый выигрыш.
